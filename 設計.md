# launchs-org システム仕様書 v5.12

> プロジェクト内・複数コンテナ構成プラットフォーム（完全自動化・GitHub連携・シームレスログ）

---

## 1. 設計思想

launchs-org は、開発者がインフラ構成を意識することなく、GitHub リポジトリから直接、フロントエンド・バックエンド・データベースなどの多層的なサービスを迅速に展開できるプラットフォームです。

- **1 Project = 1 Environment**: 1つのプロジェクト（Namespace）内で複数のコンテナを統合管理。
- **Zero-Step Build & Deploy**: コンテナ作成・更新エンドポイントを叩くだけで、ビルドからデプロイまでを自動完結。
- **GitHub-Source-of-Truth**: 各コンテナはソースコードの参照（URL/Branch/Dir）を持ち、更新時にこれらを変更して再ビルド可能。
- **オート・スナップショット**: 設定変更やデプロイのたびに、プロジェクト全体の構成を自動保存し、一括ロールバックを可能にする。
- **ゼロ・コンフィグ・ネットワーク**: 内部 IP の自動割り当てと、サブドメインの自動払い出しによる即時公開。

---

## 2. 認証・認可

- 認証方式は既存の JWT ミドルウェアを使用する（Bearer トークン）。
- `OwnerID` はリクエストの JWT クレームからユーザー ID を取得して設定する。
- アクセス制御は `OwnerID` の一致チェックのみ（マルチテナントの役割管理は現スコープ外）。

## 3. データモデル (GORM)

### 3.1 Project

```go
type Project struct {
    ID              string           `gorm:"primaryKey"`
    Name            string           `gorm:"uniqueIndex"`
    K8sResourceName string           `gorm:"uniqueIndex"`
    Namespace       string           `gorm:"uniqueIndex"`
    OwnerID         string           `gorm:"index"`
    Containers      []Container      `gorm:"foreignKey:ProjectID"`
    BuildJobs       []BuildJob       `gorm:"foreignKey:ProjectID"`
    Histories       []ProjectHistory `gorm:"foreignKey:ProjectID"`
    CreatedAt       time.Time
    DeletedAt       gorm.DeletedAt   `gorm:"index"`
}
```

### 3.2 Container

実行単位。GitHub の参照情報・ステータス・リソース制限を保持。

```go
type Container struct {
    ID            string         `gorm:"primaryKey"`
    ProjectID     string         `gorm:"index"`
    Name          string         `gorm:"index"`

    // ソース・イメージ情報
    ImageID       string         `gorm:"index"`
    RepositoryURL string         // GitHub URL
    Branch        string         `gorm:"default:'main'"`
    Directory     string         `gorm:"default:'/'"`
    Version       string         // 適用中のタグ/バージョン

    // 実行設定
    Replicas      int            `gorm:"default:1"`
    EnvVars       string         `gorm:"type:text"` // JSON: {"KEY": "VALUE"}

    // リソース制限 (v5.12 追加)
    Resources     string         `gorm:"type:text"`
    // JSON: {"requests":{"cpu":"100m","memory":"128Mi"},"limits":{"cpu":"500m","memory":"512Mi"}}

    // ステータス (v5.12 追加)
    // "Building" | "Running" | "Failed" | "Stopped"
    Status        string         `gorm:"default:'Stopped'"`

    Service       Service        `gorm:"foreignKey:ContainerID"`
    Ingress       *Ingress       `gorm:"foreignKey:ContainerID"`
    CreatedAt     time.Time
}
```

### 3.3 Service & Ingress

```go
type Service struct {
    ID          string    `gorm:"primaryKey"`
    ContainerID string    `gorm:"uniqueIndex"`
    Type        string    // "ClusterIP", "NodePort", "LoadBalancer"
    Ports       string    `gorm:"type:text"` // JSON: [{"name":"http","port":80,"target":3000}]
    InternalIP  string
    ExternalIP  string
    CreatedAt   time.Time
}

type Ingress struct {
    ID          string    `gorm:"primaryKey"`
    ContainerID string    `gorm:"uniqueIndex"`
    Subdomain   string    `gorm:"uniqueIndex"` // 自動生成: {proj-name}-{cont-name}.launchs.org
    HttpPort    int
    TlsEnabled  bool      `gorm:"default:false"` // v5.12: TLS は現スコープ外のため false
    CreatedAt   time.Time
}
```

### 3.4 Image & BuildJob

```go
type Image struct {
    ID          string    `gorm:"primaryKey"`
    ContainerID string    `gorm:"index"`
    Type        string    // "system" or "user"
    Name        string
    Registry    string
    CreatedAt   time.Time
}

type BuildJob struct {
    ID            string     `gorm:"primaryKey"`
    ProjectID     string     `gorm:"index"`
    ContainerID   string     `gorm:"index"`

    // 実行時のスナップショット情報
    RepositoryURL string
    Branch        string
    Directory     string

    // Status: "Queued" | "Running" | "Cancelled" | "Failed" | "Success"  (v5.12: Queued/Cancelled 追加)
    Status        string
    BuildLog      []byte     `gorm:"type:blob"`
    StartedAt     time.Time
    FinishedAt    *time.Time
}
```

### 3.5 ContainerLog (v5.12 追加)

稼働中コンテナの実行時ログを保持するテーブル。

```go
type ContainerLog struct {
    ID          string    `gorm:"primaryKey"`
    ContainerID string    `gorm:"index"`
    PodName     string    // ログ取得元の Pod 名
    Log         []byte    `gorm:"type:blob"`
    CollectedAt time.Time `gorm:"index"` // ローテート基準に使用
}
```

### 3.6 ProjectHistory

```go
type ProjectHistory struct {
    ID          string    `gorm:"primaryKey"`
    ProjectID   string    `gorm:"index"`
    VersionName string
    ConfigData  string    `gorm:"type:text"` // 全 Container 設定の JSON スナップショット
    CreatedAt   time.Time
}
```

## 4. 運用フロー詳細

### 4.1 コンテナ作成と自動ビルド (POST)

1. `Container` レコードを作成（Status: `Stopped`）。
2. `BuildJob` を `Queued` 状態で作成。
3. ビルドキューワーカーが順次ジョブを取り出し、K8s Job を発行してビルド・プッシュを実行。
4. ビルド成功後、K8s Deployment を作成し `Container.Status` を `Running` に更新。

### 4.2 コンテナ更新と再ビルド (PATCH)

1. 更新直前のプロジェクト構成を `ProjectHistory` に自動保存。
2. `BuildJob` を `Queued` 状態で作成（同一コンテナの既存 Queued/Running ジョブの後ろに積む）。
3. ビルド成功後、`Container.Version` に新イメージタグを反映し K8s ローリングアップデートを実行。

### 4.3 ビルドキューイングとキャンセル

- キュー実装: `BuildJob` テーブルを DB ポーリングで監視。`Status = Queued` のジョブを `ContainerID` 単位で FIFO 処理。
- 同一コンテナに対する同時実行は1件のみ。後続は `Queued` で待機。
- キャンセル: `POST /v1/build-jobs/:id/cancel` を受け付け、対応する K8s Job を削除し `Status` を `Cancelled` に更新。
  - `Queued` 状態のジョブはそのまま `Cancelled` に変更（K8s Job 削除不要）。
  - `Running` 状態のジョブは K8s Job を削除してから `Cancelled` に変更。

### 4.4 ビルド失敗時の挙動

- `Container.Status` を `Failed` に更新。
- K8s Deployment は直前の成功イメージのまま維持（ロールアウトしない）。
- `BuildJob.BuildLog` にエラーログを保存。

### 4.5 シームレス・ログストリーミング（ビルドログ）

- SSE 接続の初回メッセージで `BuildJob.BuildLog` の全データを送信し、以降をリアルタイム追従配信。
- 再接続時も全ログを確認可能。
- ログは K8s Job の Pod ログをバックエンドがリアルタイムで取得し、逐次 `BuildJob.BuildLog` に追記する。

### 4.6 コンテナ実行ログ取得 (v5.12 追加)

- 稼働中コンテナのログは `GET /v1/containers/:id/logs` で取得可能。
- SSE によるリアルタイムストリーミングと、過去ログ（`ContainerLog` テーブル）の一括取得の両方をサポート。
- バックエンドは `client-go` を使って対象 Pod のログを Watch し、`ContainerLog` テーブルに逐次保存する。
- 複数 Replica が存在する場合は全 Pod のログを集約して返す。

### 4.7 ログローテート (v5.12 追加)

ビルドログ・コンテナログともに下記のローテートポリシーを適用する。

| 対象 | 保持期間 | 処理方式 |
|------|----------|----------|
| `BuildJob.BuildLog` | 30日 | バッチで `FinishedAt` が 30日以上前のレコードの `BuildLog` を nil クリア |
| `ContainerLog` | 7日 | バッチで `CollectedAt` が 7日以上前のレコードを物理削除 |

- ローテートバッチは Go の `time.Ticker` によりサーバー起動時にバックグラウンドで定期実行（デフォルト: 毎日 UTC 03:00）。
- ローテート実行間隔・保持期間は環境変数で設定可能とする。

```
LOG_ROTATE_INTERVAL_HOURS=24
BUILD_LOG_RETENTION_DAYS=30
CONTAINER_LOG_RETENTION_DAYS=7
```

### 4.8 ロールバック

1. `ProjectHistory.ConfigData` から全 Container 設定を復元。
2. 各 Container に対して `BuildJob` を `Queued` で発行し、保存済みイメージタグで K8s をローリングアップデート。
3. ロールバック中も既存 Deployment は稼働し続ける。

### 4.9 サブドメイン自動生成

Ingress 有効化時、`{project-name}-{container-name}.launchs.org` を自動割り当て。TLS は現スコープ外。

## 5. API 仕様書

### 5.1 プロジェクト (Projects)

| Method | Path | 説明 |
|--------|------|------|
| GET | `/v1/projects` | プロジェクト一覧取得 |
| POST | `/v1/projects` | プロジェクト作成（Namespace 初期化） |
| GET | `/v1/projects/:id` | プロジェクト詳細取得 |
| DELETE | `/v1/projects/:id` | プロジェクト削除（K8s Namespace + 全リソース含むハードデリート） |

### 5.2 コンテナ & 自動ビルド (Containers & Build)

| Method | Path | 説明 |
|--------|------|------|
| POST | `/v1/projects/:id/containers` | コンテナ作成・自動ビルド開始 |
| PATCH | `/v1/containers/:id` | コンテナ更新・再ビルド実行 |
| DELETE | `/v1/containers/:id` | コンテナ削除（K8s Deployment + 関連リソース含むハードデリート） |
| GET | `/v1/containers/:id/build-jobs` | 過去のビルド履歴一覧 |
| GET | `/v1/stream/build-jobs/:id` | SSE: 過去分＋リアルタイムビルドログストリーム |

**POST /v1/projects/:id/containers Body**
```json
{
  "name": "string",
  "repository_url": "string",
  "branch": "string",
  "directory": "string",
  "env_vars": {"KEY": "VALUE"},
  "replicas": 1,
  "resources": {
    "requests": {"cpu": "100m", "memory": "128Mi"},
    "limits": {"cpu": "500m", "memory": "512Mi"}
  }
}
```

**PATCH /v1/containers/:id Body**
```json
{
  "repository_url": "string (optional)",
  "branch": "string (optional)",
  "directory": "string (optional)",
  "env_vars": {"KEY": "VALUE (optional)"},
  "replicas": 1,
  "resources": {
    "requests": {"cpu": "100m", "memory": "128Mi"},
    "limits": {"cpu": "500m", "memory": "512Mi"}
  }
}
```

### 5.3 ビルドジョブ管理 (Build Jobs)

| Method | Path | 説明 |
|--------|------|------|
| POST | `/v1/build-jobs/:id/cancel` | ビルドキャンセル（Queued/Running 両対応） |

### 5.4 コンテナログ (Container Logs) (v5.12 追加)

| Method | Path | 説明 |
|--------|------|------|
| GET | `/v1/containers/:id/logs` | 過去ログ一覧取得（`ContainerLog` テーブルから） |
| GET | `/v1/stream/containers/:id/logs` | SSE: 過去分＋リアルタイムログストリーム |

### 5.5 ネットワーク (Networking)

| Method | Path | 説明 |
|--------|------|------|
| PATCH | `/v1/containers/:id/service` | 内部ポート設定変更 |
| POST | `/v1/containers/:id/ingress` | Ingress 有効化（サブドメイン自動払い出し） |
| DELETE | `/v1/containers/:id/ingress` | 外部公開停止 |

### 5.6 履歴 (History)

| Method | Path | 説明 |
|--------|------|------|
| GET | `/v1/projects/:id/histories` | 自動スナップショット一覧 |
| POST | `/v1/projects/:id/rollback/:phid` | 特定の履歴 ID への復元 |

## 6. エラーレスポンス仕様

全エンドポイント共通のエラーフォーマット。

```json
{
  "code": "NOT_FOUND",
  "message": "container not found"
}
```

主要エラーコード一覧：

| HTTP Status | code | 説明 |
|-------------|------|------|
| 400 | `BAD_REQUEST` | リクエストパラメータ不正 |
| 401 | `UNAUTHORIZED` | JWT 認証失敗 |
| 403 | `FORBIDDEN` | 他ユーザーのリソースへのアクセス |
| 404 | `NOT_FOUND` | リソースが存在しない |
| 409 | `CONFLICT` | 名前の重複など |
| 500 | `INTERNAL_ERROR` | サーバー内部エラー |

## 7. Container.Status 遷移図

```
[作成]
  │
  ▼
Stopped ──── PATCH/POST ────► Building
  ▲                               │
  │                         ビルド成功
  │                               │
  │                               ▼
  │                            Running
  │                               │
  │                         ビルド失敗 or K8s 異常
  │                               │
  └───── rollback ◄────────── Failed
                    DELETE ──► (レコード削除)
```

## 8. インフラ・技術スタック

| 項目 | 採用技術・方針 |
|------|---------------|
| K8s 操作 | `client-go` を直接使用 |
| ビルド実行 | K8s Job として実行（ビルド・プッシュまでのコードは別途実装済み） |
| イメージレジストリ | 自前プライベートレジストリ |
| GitHub | パブリックリポジトリのみ対応 |
| TLS | 現スコープ外（`Ingress.TlsEnabled = false`） |
| ビルドキュー | `BuildJob` テーブルの DB ポーリング方式 |
| ログローテート | Go `time.Ticker` によるバックグラウンドバッチ |
