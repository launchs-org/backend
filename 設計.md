## 1. 設計思想

ACP は Kubernetes の複雑なリソース管理を抽象化し、開発者が **「環境 (Deployment)」** と **「成果物 (Container)」** を分離して管理できるプラットフォームである。Railway の UX を参考に、インフラを意識せずデプロイできる体験を目指す。

| 原則 | 内容 |
|------|------|
| **不変の接続性** | Deployment ID を固定し、外部・内部からのアクセスポイントを不変にする |
| **アジャイルな更新** | Container 単位でバージョン管理し、高速ビルド・デプロイと世代管理を実現する |
| **マルチプロジェクト分離** | Project ごとに K8s Namespace を分け、テナント間のリソース競合を防ぐ |
| **Railway ライク** | 環境変数・ポート・レプリカ数を Deployment 単位で管理し、直感的な操作を実現する |

## 2. リソース構造とライフサイクル

### 2.1 階層構造

```
Project (UUID)
├── Namespace: acp-{project_uuid[0:12]}
├── Shared Volumes
└── Deployment (ユーザー入力 ID、プロジェクト内一意)
    ├── EnvVars (Deployment に紐づく)
    ├── Ports (複数対応)
    ├── Replicas
    ├── Service / Ingress (K8s リソース名で固定)
    └── Container (UUID、ビルドごと、最新3世代保持)
        ├── Image (Harbor URL, tag=UUID)
        ├── BuildLog (圧縮)
        └── Status: Building → Success / Failed → Retired
```

### 2.2 リソース一覧

| コンポーネント | 識別子 | 生存期間 | 備考 |
|------------|--------|---------|------|
| Project | UUID | ユーザー削除まで | K8s Namespace・共有 Volume を管理 |
| Deployment | ユーザー入力文字列 (プロジェクト内一意) | 削除まで不変 | Service/Ingress の接続先。Container を入れ替えるシェル |
| Container | UUID (ビルドごと) | 最新 3 世代保持 | イメージタグ・ビルドログ・ステータスを保持 |
| Volume | UUID | Project 削除まで | RWX モード。Deployment 間で共有可能 |

### 2.3 Deployment ID の制約

Deployment ID はユーザーが自由文字列で入力する。K8s リソース名は RFC 1123 に準拠する必要があるため、**入力バリデーション** と **K8s リソース名の別途生成** を行う。

**バリデーション規則**

```
正規表現: ^[a-z][a-z0-9-]{0,61}[a-z0-9]$
- 英小文字・数字・ハイフンのみ
- 先頭は英小文字
- 63文字以内
```

**K8s リソース名の生成ルール**

```
k8s_resource_name = {project_id[0:8]}-{deployment_id}
例: a3f2c1bd-auth-server

- deployment_k8s_name テーブルでマッピングを管理
- 一度発行した k8s_resource_name は変更不可（Service の DNS が変わるため）
- プロジェクト間で同一 Deployment ID が使われても衝突しない
```

### 2.4 K8s Namespace 戦略

| 用途 | Namespace |
|------|-----------|
| ACP システム自身 (Auth/Build/Deploy) | `acp-system` |
| 各 Project のワークロード | `acp-{project_uuid[0:12]}` |

- Project 1つ = K8s Namespace 1つ
- Project 削除時: §6.4 の手順で個別リソースを削除してから Namespace を削除する

## 3. データモデル (GORM Schema)

```go
// Project
type Project struct {
    ID        string    `gorm:"primaryKey"`          // UUID
    Name      string
    Namespace string    `gorm:"uniqueIndex"`          // K8s Namespace 名
    OwnerID   string    `gorm:"index"`
    CreatedAt time.Time
}

// Deployment — ユーザー入力 ID と K8s リソース名を分離
type Deployment struct {
    ID                string    `gorm:"primaryKey"`   // ユーザー入力 (例: "auth-server")
    ProjectID         string    `gorm:"index"`
    K8sResourceName   string    `gorm:"uniqueIndex"`  // K8s 用生成名
    ActiveContainerID string
    IsDomainIssued    bool      `gorm:"default:false"`
    Replicas          int       `gorm:"default:1"`
    CreatedAt         time.Time
    Containers        []Container   `gorm:"foreignKey:DeploymentID"`
    EnvVars           []EnvVar      `gorm:"foreignKey:DeploymentID"`
    Ports             []Port        `gorm:"foreignKey:DeploymentID"`
}

// Container — ビルドごとのバージョン実体
type Container struct {
    ID           string    `gorm:"primaryKey"`        // UUID
    DeploymentID string    `gorm:"index"`
    ProjectID    string    `gorm:"index"`             // GC・削除時の効率化
    Image        string                               // Harbor URL (tag=UUID)
    Status       string    // Building / Success / Failed / Retired
    BuildLog     []byte    `gorm:"type:blob"`         // 圧縮ログ
    CreatedAt    time.Time
}

// EnvVar — Deployment に紐づく環境変数 (Railway 方式)
type EnvVar struct {
    ID           string `gorm:"primaryKey"`
    DeploymentID string `gorm:"index"`
    Key          string
    Value        string // 平文保存 or 暗号化 (§9.1 参照)
}

// Port — Deployment に紐づく公開ポート (複数対応)
type Port struct {
    ID           string `gorm:"primaryKey"`
    DeploymentID string `gorm:"index"`
    Port         int
    Protocol     string `gorm:"default:TCP"` // TCP / UDP
    Description  string // 任意メモ (例: "http", "metrics")
}
```

> **⚠ Container.Status のステータス遷移**
>
> ```
> Building → Success → (Active として稼働) → Retired (GC で削除)
>          ╰→ Failed
> ```
> 元仕様には `Building` が欠落していた。ビルド中の状態追跡に必須のため追加。

## 4. マイクロサービス構成

### 4.1 サービス一覧

| サービス | Namespace | 役割 | 外部公開 |
|--------|-----------|------|---------|
| **Auth** | acp-system | JWT 発行 (ユーザー認証) / 内部トークン発行 (サービス間認証) | あり |
| **Build** | acp-system | Railpack でビルド・Harbor push・DB 記録・Deploy Svc 呼び出し | あり |
| **Deploy** | acp-system | client-go で K8s リソース操作・昇格・ロールバック・GC | あり |
| **Database** | acp-system | PostgreSQL (スキーマ別マルチサービス) | なし |

### 4.2 サービス間通信

各サービスは `.env` で指定された内部 FQDN を通じて gRPC で通信する。
外部からのリクエストは、各サービスの Ingress を経由して直接到達する。

```
Client      ──(HTTP/JWT)──▶ Auth (ログイン)
Client      ──(HTTP/JWT)──▶ Build (ビルド開始)
Build       ──(gRPC)──────▶ Deploy   ← ビルド完了後の昇格リクエスト
Deploy      ──(client-go)─▶ K8s API
```

## 5. API 仕様 (REST)

### 5.0 共通仕様

**ベース URL**
```
https://api.domain.org/v1
```

**共通リクエストヘッダ**

| ヘッダ | 必須 | 値 |
|--------|------|----|
| `Authorization` | ✅ | `Bearer {jwt}` |
| `Content-Type` | POST/PUT/PATCH のみ | `application/json` |
| `Accept` | 任意 | `application/json` |

**共通レスポンス形式**

成功時:
```json
{
  "data": { ... },
  "meta": {
    "request_id": "uuid"
  }
}
```

エラー時:
```json
{
  "error": {
    "code": "DEPLOYMENT_NOT_FOUND",
    "message": "deployment 'auth-server' not found in project 'xxx'",
    "details": {}
  },
  "meta": {
    "request_id": "uuid"
  }
}
```

**共通エラーコード**

| HTTP | code | 説明 |
|------|------|------|
| 400 | `VALIDATION_ERROR` | バリデーション失敗 |
| 401 | `UNAUTHORIZED` | JWT 未送付・期限切れ |
| 403 | `FORBIDDEN` | 操作権限なし |
| 404 | `NOT_FOUND` | リソースが存在しない |
| 409 | `CONFLICT` | 既に同名リソースが存在する |
| 422 | `UNPROCESSABLE` | 論理的に処理不可 (GC済みへのロールバックなど) |
| 500 | `INTERNAL_ERROR` | サーバー内部エラー |

### 5.4 ステータスポーリング

フロントが一定間隔（推奨: **3秒**）で叩いてデプロイ進捗を同期するエンドポイント。

#### `GET /projects/:project_id/deployments/:id/status` — Deployment ステータス取得

**Deployment.status の定義**

| status | 意味 | 終端? |
|--------|------|-------|
| `idle` | 一度もデプロイされていない | — |
| `building` | Railpack でイメージビルド中 | — |
| `pushing` | Harbor へイメージ push 中 | — |
| `deploying` | K8s Deployment を Patch 中 | — |
| `image_pulling` | K8s が Harbor からイメージ pull 中 | — |
| `pod_pending` | Pod がスケジューリング待ち | — |
| `running` | 全 Pod が Ready | ✅ 安定 |
| `failed` | ビルドまたはデプロイが失敗 | ✅ 終端 |
| `crashed` | Pod が起動後に異常終了 (CrashLoopBackOff) | ✅ 終端 |

**ステータス遷移**
```
idle
 └─▶ building ─▶ pushing ─▶ deploying ─▶ image_pulling ─▶ pod_pending ─▶ running
                                                                              ↕
       failed ◀─────────────────────────────── (各フェーズで失敗)         crashed
```

**Response `200`**
```json
{
  "data": {
    "deployment_id": "auth-server",
    "status": "pod_pending",
    "active_container_id": "uuid-of-current-build",
    "replicas": {
      "desired": 2,
      "ready": 0,
      "available": 0
    },
    "pods": [
      {
        "name": "auth-server-7d9f-xk2p",
        "status": "Pending",
        "ready": false,
        "restart_count": 0,
        "message": "0/3 nodes are available: insufficient memory"
      },
      {
        "name": "auth-server-7d9f-mn4q",
        "status": "Pending",
        "ready": false,
        "restart_count": 0,
        "message": "0/3 nodes are available: insufficient memory"
      }
    ],
    "last_event": {
      "type": "Warning",
      "reason": "FailedScheduling",
      "message": "0/3 nodes are available: insufficient memory",
      "occurred_at": "2025-01-01T00:00:05Z"
    },
    "checked_at": "2025-01-01T00:00:10Z"
  }
}
```

**フロント側ポーリング実装ガイドライン**

```
- ポーリング間隔: 3秒
- 停止条件: status が "running" / "failed" / "crashed" になったら停止
- タイムアウト: 10分経過しても終端状態にならない場合はポーリングを停止し
               ユーザーに警告を表示する
- crashed 判定: pods[].restart_count >= 3 かつ pods[].status == "CrashLoopBackOff"
```

## 6. WebSocket 仕様

### 6.0 共通仕様

**ベース URL**
```
wss://api.launchs.org/v1/ws
```

**認証**

WebSocket はハンドシェイク時に HTTP ヘッダで JWT を送る。接続確立後のトークン更新は非対応（再接続で対応）。

```
GET wss://api.launchs.org/v1/ws/...
Headers:
  Authorization: Bearer {jwt}
  Upgrade: websocket
  Connection: Upgrade
```

**接続ライフサイクル**

```
クライアント                          サーバー
    │                                    │
    │── Upgrade: websocket ──────────────▶│  ハンドシェイク (JWT 検証)
    │◀── 101 Switching Protocols ─────────│
    │                                    │
    │◀── { type: "connected" } ──────────│  接続確立通知
    │                                    │
    │◀── { type: "log", ... } ───────────│  ログ/イベント配信
    │◀── { type: "log", ... } ───────────│
    │                                    │
    │◀── { type: "done" } ───────────────│  完了通知
    │                                    │
    │  (クライアントが close frame を送る)  │
    │── close ──────────────────────────▶│
    │◀── close ───────────────────────────│
```

**サーバー側タイムアウト**

| 種別 | 時間 | 挙動 |
|------|------|------|
| アイドルタイムアウト | **5分** | メッセージ送受信がない場合、サーバーが close frame を送って切断 |
| ビルド最大時間 | **30分** | ビルドが終わらない場合、`type: "timeout"` を送って切断 |
| ping/pong | **30秒** | サーバーが ping を送り、応答なければ切断 |

**クライアント側再接続ガイドライン**

```
- close code 1000 (正常終了): 再接続しない
- close code 1001/1006/1011 (異常切断): 指数バックオフで再接続
  例: 1秒 → 2秒 → 4秒 → 8秒 → 最大30秒
- 再接続時: 同じ URL に再度接続するとログの続きから配信される
  (サーバーはクライアントが最後に受信したログ行を cursor パラメータで受け取る)
- type: "done" を受信後はクライアントから close frame を送って終了する
```

**サーバー → クライアント メッセージ形式**

```json
{

## 7. 運用フロー

### 7.1 ビルド〜自動昇格フロー

```
User
 │  POST /builds
 ▼
Build Svc
 ├─ 1. Container レコード作成 (Status=Building)
 ├─ 2. Railpack でイメージビルド
 ├─ 3. Harbor へ push (tag={container_uuid})
 ├─ 4. Container.Status = Success
 └─ 5. gRPC: Deploy Svc.PromoteContainer(container_id)
           │
           ▼
        Deploy Svc
         ├─ 6. K8s Deployment の image を新 UUID に Patch
         │      (EnvVars・Ports・Replicas も同時に適用)
         ├─ 7. Deployment.ActiveContainerID 更新
         └─ 8. GC 実行 (4世代目以降を Harbor・DB から削除)
```

**失敗時の挙動**

| ステップ | 失敗時 |
|---------|--------|
| 2. ビルド失敗 | `Status=Failed` で終了。K8s は変更されない |
| 6. K8s Patch 失敗 | `ActiveContainerID` を旧値のまま保持。Deploy Svc がエラーを返す |
| 8. GC 失敗 | ログ記録してスキップ。次回ビルド時に再試行 |

### 7.2 手動ロールバック

自動ロールバックは実装しない。ユーザーが明示的に旧 Container UUID を指定して操作する。

```
POST /deployments/:id/rollback
  body: { "container_id": "<old_uuid>" }

Deploy Svc:
  1. Container.Status が Success であることを確認
  2. Harbor にイメージが存在することを確認
  3. K8s Deployment の image を指定 UUID に Patch
  4. Deployment.ActiveContainerID を更新
```

> GC で Retired になった Container へのロールバックは不可。エラーを返す。

### 7.3 世代管理 (GC)

- **トリガー**: Deploy Svc がビルド昇格を完了したタイミング
- **対象**: 同一 `DeploymentID` 内で `CreatedAt` が古い順に 4 番目以降
- **手順**:
  1. 対象 Container の `Status` を `Retired` に更新
  2. Harbor から該当タグのイメージを削除
  3. DB から Container レコードを削除
- **失敗時**: ログ記録のみ。次回ビルド時に再試行（孤立イメージは定期クリーンアップジョブでもカバー）

### 7.4 Project 削除フロー

> K8s を先に消すと Harbor に孤立イメージが残るため、以下の順序を厳守する。

```
1. DB: Project 配下の全 Deployment・Container を論理削除 (deleted_at 付与)
2. Harbor: 全 Container の image タグを削除 (並列処理可)
3. K8s: Namespace を delete → Pod・Service・Ingress・Volume が連鎖削除
4. DB: 論理削除レコードを物理削除
```

各ステップは冪等に設計し、失敗時はリトライ可能とする。

## 8. ドメイン・ルーティング

### 8.1 内部通信 (固定 FQDN)

```
Service 命名: svc-{k8s_resource_name}
FQDN:        svc-{k8s_resource_name}.{namespace}.svc.cluster.local

例: svc-550e8400-auth-server.acp-550e8400e29b.svc.cluster.local

Selector:    app={k8s_resource_name}
             ↑ 常に Active Container の Pod が持つラベル
```

Deployment ID が変わらない限り FQDN も不変。Deployment 作成レスポンスの `internal_fqdn` フィールドで確認できる。

### 8.2 外部公開ドメイン

| 条件 | 内容 |
|------|------|
| `IsDomainIssued = false` | 外部公開なし |
| `IsDomainIssued = true` | `{deployment_id}.{project_id}.launchs.org` を自動発行 |

- Wildcard DNS `*.launchs.org` で解決
- Ingress がリクエストを `svc-{k8s_resource_name}` へプロキシ
- 複数ポートを公開する場合、ポートごとに Ingress ルールを生成

### 8.3 サブドメイン禁止リスト

`*.launchs.org` のサブドメインとして使用できない予約語を設定ファイルで管理する。

**管理方式**

```yaml
# config/blocked_subdomains.yaml (または環境変数 BLOCKED_SUBDOMAINS=www,api,admin,... )
blocked_subdomains:
  - www
  - api
  - admin
  - root
  - mail
  - smtp
  - ftp
  - ssh
  - dashboard
  - status
  - docs
  - support
  - billing
  - acp
  - system
```

**バリデーションタイミング**

```
Deployment 作成時 (POST /deployments):
  1. deployment_id が blocked_subdomains に含まれていないか確認
  2. 含まれている場合: 409 CONFLICT を返す
     {
       "error": {
         "code": "SUBDOMAIN_RESERVED",
         "message": "'api' is a reserved subdomain and cannot be used as a deployment ID",
         "details": {}
       }
     }
```

> 禁止リストはサービス再起動なしに反映できるよう、起動時にファイルをロードして
> メモリ上に保持する設計を推奨。変更時はサービス再起動が必要。

## 9. 環境変数・ポート・スケーリング

### 8.1 環境変数 (Railway 方式)

環境変数は **Deployment に紐づける**。ビルド時・デプロイ時どちらでも参照可能。暗号化は行わず DB に平文保存とする。

```
管理単位: Deployment (Container ごとではない)
暗号化:   なし（DB 平文保存）
反映タイミング:
  - 新規ビルド時: ビルドコンテキストに渡す (Railpack の env 設定)
  - ENVのみ変更時: 新規ビルドなしで K8s Deployment を Patch して即時反映
                   → Deployment 全体の Rolling Restart が発生する
```

**ENVのみ更新フロー (リビルドなし)**

```
PUT /deployments/:id/env
  │
  ▼
Deploy Svc
  ├─ DB の EnvVar レコードを一括更新
  ├─ K8s Deployment の env を Patch
  └─ K8s が Rolling Restart を実行（全 Pod が順次再起動）

レスポンス:
  {
    "updated": true,
    "rolling_restart": true   ← クライアントへの通知フラグ
  }
```

> **Rolling Restart の挙動**
> - 対象: Deployment 配下の**全 Pod**（replicas 数にかかわらず）
> - 方式: K8s の Rolling Update（`maxUnavailable: 0` がデフォルト推奨。ダウンタイムなし）
> - クライアントへの通知: レスポンスの `rolling_restart: true` フラグのみ。UI 側でバナー表示などを行う
> - ユーザーへの周知: API ドキュメントに「ENV 更新は即時 Rolling Restart を伴う」と明記する

> **⚠ Secret 管理 (将来対応)**
> K8s Secret との同期は今後の拡張ポイント (§13) に記載。

### 8.2 ポート設定 (複数対応)

```
複数ポートをサポート。各ポートに対して K8s Service の port エントリを生成する。

例: Port テーブルのレコード
  { port: 3000, protocol: TCP, description: "http"    }
  { port: 9090, protocol: TCP, description: "metrics" }

→ K8s Service:
  ports:
    - name: http    port: 3000 targetPort: 3000
    - name: metrics port: 9090 targetPort: 9090
```

### 8.3 スケーリング

```
デフォルト: replicas = 1
API での変更: PATCH /deployments/:id/replicas  body: { "replicas": 3 }

Deploy Svc:
  K8s Deployment の spec.replicas を即時 Patch
  → DB の Deployment.Replicas も更新
```

> HPA (Horizontal Pod Autoscaler) は今後の拡張ポイント (§13) に記載。

## 10. 認証・認可

### 9.1 2層認証モデル

| 層 | 対象 | 仕組み |
|----|------|--------|
| **Layer 1: ユーザー認証** | ユーザー → 各サービス | Auth Svc が JWT を発行。各サービス（Auth/Build/Deploy）のミドルウェアで検証 |
| **Layer 2: サービス間認証** | 各 Svc → 各 Svc | Auth Svc が内部トークンを発行。gRPC メタデータで送付 |

### 9.2 サービス間認証の詳細

```
- 各サービスが起動時に Auth Svc から内部トークンを取得
- gRPC メタデータ: authorization: Bearer <internal-token>
- トークン有効期限: 1時間（自動更新）
- NetworkPolicy: acp-system 内の Pod 間通信のみ許可。Project Namespace からは遮断
```

> **⚠ TLS なし (NetworkPolicy のみ)**
> 通信は暗号化されない。クラスター外へのルーティングは絶対に通さない設計とする。
> mTLS 対応は今後の拡張ポイント (§13) に記載。

## 12. 元仕様からの変更点・解決した不整合

| # | 項目 | 元仕様 | 確定仕様 | 理由 |
|---|------|--------|---------|------|
| 1 | Container.Status | Success / Failed / Retired | **Building を追加** (4状態) | ビルド中の状態が追跡不能だった |
| 2 | Build → Deploy 連携 | DBイベント保持 (曖昧) | **Build Svc が Deploy Svc に gRPC で直接呼び出し** | ポーリング遅延・複雑さを避ける |
| 3 | ロールバック | 自動 (Pod 監視) | **ユーザーが手動で API 呼び出し** | 自動ロールバックの誤発動リスクが高い |
| 4 | K8s リソース名 | Deployment ID をそのまま使用 | **`{project_id}-{deployment_id}` で生成しマッピングテーブル管理** | プロジェクト間で同一 ID が衝突する |
| 5 | Deployment ID 制約 | 未定義 | **英小文字・数字・ハイフン・63文字以内** | K8s RFC 1123 制約に合わせる |
| 6 | サービス間認証 | 未定義 | **Auth Svc が内部トークンを発行・gRPC メタで検証** | NetworkPolicy だけでは認可ができない |
| 7 | Project 削除順序 | 未定義 | **DB論理削除 → Harbor削除 → K8s Namespace削除 → DB物理削除** | 孤立リソースを防ぐ |
| 8 | 環境変数管理 | 未定義 | **Deployment に紐づける (Railway 方式)** | ENVのみ更新でリビルド不要にする |
| 9 | ポート設定 | 未定義 | **複数ポート対応 (Port テーブルで管理)** | HTTP + metrics など複数ポートのユースケース |
| 10 | スケーリング | 未定義 | **Deployment.Replicas、API で増減** | デフォルト1、必要に応じてスケールアウト |
| 11 | ENV 暗号化 | 未定義 | **暗号化なし（DB 平文保存）** | 実装コスト優先。Secret 管理は将来対応 |
| 12 | Rolling Restart 通知 | 未定義 | **レスポンスに `rolling_restart: true` を返却。API ドキュメントに明記** | ユーザーが再起動を認識できるようにする |

## 13. 今後の拡張ポイント

- **Preview URL**: Active 以外の Container UUID を指す一時的な Service/Ingress をオンデマンド生成
- **Secret 管理**: K8s Secret と抽象化環境変数の同期
- **HPA 対応**: CPU/メモリ使用率に基づく自動スケーリング
- **mTLS**: サービス間通信の暗号化 (Istio / Linkerd 導入)
- **リソース監視**: Deployment 単位のメトリクス取得 (Prometheus / Grafana)
- **CI/CD 連携**: Git push フックによる自動ビルドトリガー
- **自動ロールバック**: Readiness/Liveness Probe の結果を見た自動ロールバック
- **マルチクラスター**: 複数 K8s クラスターへの分散デプロイ
